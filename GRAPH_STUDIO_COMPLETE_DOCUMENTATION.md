## 📝 Описание

Graph Studio представляет собой **универсальный редактор и визуализатор графов связей**.

Граф состоит из двух базовых сущностей:

1. **Узел** — объект, понятие, группа, ядро или акцентный элемент.
2. **Связь** — отношение между двумя узлами.

Программа позволяет вручную создать структуру, настроить её внешний вид, увидеть результат в реальном времени, сохранить проект в JSON и встроить готовый граф на другой сайт.

Проект не привязан к магазину, товарам, рискам, положительным или отрицательным свойствам. Семантика графа полностью определяется пользователем. Один и тот же движок подходит для:

- структуры проекта;
- карты знаний;
- организационной схемы;
- зависимостей программных модулей;
- причинно-следственных связей;
- архитектуры информационной системы;
- учебной схемы;
- графа сущностей;
- карты заинтересованных сторон;
- визуализации процессов;
- связей между документами;
- любой другой сетевой структуры.

Главный принцип проекта:

> **Данные, визуализация, интерфейс, машинные команды и транспорт MCP разделены на независимые модули.**

Благодаря этому граф можно использовать целиком, частично или встроить в существующую систему без копирования административной панели.

## 🎯 Для чего нужна программа

Graph Studio решает несколько практических задач.

### 1. Ручное создание графа

Администратор добавляет узлы и связи через формы, меняет цвета, размеры, подписи, расположение и анимацию.

### 2. Визуальное исследование структуры

Пользователь вращает граф, масштабирует сцену, наводит курсор на узлы и получает подробную информацию.

### 3. Подготовка презентационного модуля

После настройки административный интерфейс можно скрыть и встроить только готовый граф.

### 4. Построение графа из данных ИИ

Модель формирует простой JSON формата `graph-studio/1`, после чего приложение проверяет данные и воспроизводит структуру.

### 5. Управление через атомарные команды

ИИ, скрипт или внешний интерфейс может выполнять понятные операции:

```text
add_node
add_connection
change_node
change_connection
set_layout
change_view
```

### 6. Управление через MCP

При необходимости локальный MCP-sidecar предоставляет те же команды как инструменты, ресурсы и prompts.

## ✨ Ключевые функции и возможности

### 🕸️ Редактор структуры

- добавление узлов;
- изменение узлов;
- удаление узлов вместе с принадлежащими им связями;
- добавление связей;
- изменение подписи, цвета, описания и толщины связи;
- удаление связи;
- запрет дублирования связи между одной парой узлов;
- импорт и экспорт JSON.

### 🎨 Настройка внешнего вида

- цвета всех типов узлов;
- индивидуальный цвет узла;
- индивидуальный размер узла;
- цвет и сила связи;
- подписи только ядер или всех узлов;
- легенда;
- фоновые точки;
- частицы глубины;
- информационные карточки.

### 🪐 Две раскладки

- **Планетарная паутина:** ветви расходятся от центрального ядра.
- **Желейные соты:** узлы размещаются в шестигранной сетке и реагируют на импульсы как упругая система.

### 🧲 Интерактивная физика

- инерция камеры;
- плавное вращение;
- плавное масштабирование;
- пружины между узлами;
- притяжение к исходным позициям;
- отталкивание;
- предотвращение наложений;
- передача импульса соседним узлам;
- перетаскивание узла;
- желейная реакция структуры.

### ❤️ Органическая пульсация

- двойной сердечный удар;
- волна от ядра по уровням графа;
- изменение размера и формы узлов;
- изменение ширины связи;
- каплевидный импульс по ветви;
- возможность полностью отключить свечение.

### 💨 Потоковые частицы

- движутся непосредственно по связи;
- имеют форму вытянутого штриха;
- имеют прозрачный хвост;
- не выглядят как дополнительные узлы;
- количество, скорость, длина и толщина настраиваются.

### 📐 Адаптивная толщина связи

Сильная связь может быть толстой в центре, но автоматически сужается около маленького узла. Поэтому визуальная сила отношения сохраняется, не перекрывая конечный элемент.

### 🔒 Режим администратора и просмотра

- администратор редактирует граф;
- пользователь видит только модуль;
- формы редактирования можно полностью скрыть;
- методы записи блокируются в viewer-режиме;
- вращение, масштабирование и карточки остаются доступными.

### 🤖 Машинно-понятный контракт

- строгий JSON-план;
- 13 атомарных команд;
- одинаковые имена команд в браузере, Web Component, `postMessage` и MCP;
- JSON Schema;
- предсказуемые ошибки;
- проверка уникальности ID и связей.

## 🏗️ Архитектура

Проект построен слоями.

```text
┌──────────────────────────────────────────────────────────┐
│                         UI                               │
│ app-controller · node-info · link-editor · legend       │
└───────────────────────────┬──────────────────────────────┘
                            │ публичные методы и события
┌───────────────────────────▼──────────────────────────────┐
│                     Graph3DEngine                        │
│ состояние · камера · физика · рендер · интеракция        │
└──────────────┬────────────┬────────────┬─────────────────┘
               │            │            │
       ┌───────▼──────┐ ┌───▼──────┐ ┌──▼─────────────┐
       │ Layouts      │ │ Math     │ │ Render helpers │
       │ planetary    │ │ rotation │ │ ribbon/blob    │
       │ hex          │ │ project  │ │ flow           │
       └──────────────┘ └──────────┘ └────────────────┘
               │
┌──────────────▼───────────────────────────────────────────┐
│ Core: config · schema · utils · deterministic contracts │
└──────────────────────────────────────────────────────────┘

Отдельно:

model → AI JSON и атомарные команды
embed → Web Component и mount-функция
mcp   → необязательный транспорт к тому же словарю команд
```

### Правило зависимостей

Нижний слой не должен знать о верхнем.

Например:

- `projection.js` не ищет элементы HTML;
- `heart-pulse.js` не знает цвет узла;
- `Graph3DEngine.js` не строит формы;
- MCP-сервер не рисует Canvas;
- UI не рассчитывает физические силы самостоятельно.

Это позволяет тестировать математику отдельно и уменьшает вероятность того, что изменение кнопки разрушит раскладку. Человеческая цивилизация уже достаточно страдала от обработчиков `onclick`, содержащих половину приложения.

## 🧠 Модель данных

### Полный проект

```json
{
  "version": "8.0.0",
  "data": {
    "nodes": [],
    "links": []
  },
  "config": {}
}
```

### Узел

```json
{
  "id": "core",
  "name": "Главная тема",
  "type": "core",
  "description": "Подробное описание",
  "color": "#263846",
  "size": 25
}
```

| Поле | Тип | Обязательное | Назначение |
|---|---|---:|---|
| `id` | string | да | Уникальный стабильный идентификатор |
| `name` | string | рекомендуется | Отображаемое название |
| `type` | string | нет | `core`, `group`, `node`, `accent`, `default` |
| `description` | string | нет | Текст информационной карточки |
| `color` | string | нет | Индивидуальный цвет `#RRGGBB` |
| `size` | number | нет | Индивидуальный радиус узла |

### Связь

```json
{
  "source": "core",
  "target": "branch_1",
  "label": "содержит",
  "description": "Описание отношения",
  "color": "#8699a7",
  "width": 4
}
```

| Поле | Тип | Обязательное | Назначение |
|---|---|---:|---|
| `source` | string | да | ID первого узла |
| `target` | string | да | ID второго узла |
| `label` | string | нет | Краткая подпись отношения |
| `description` | string | нет | Подробное описание |
| `color` | string | нет | Индивидуальный цвет |
| `width` | number | нет | Центральная сила/толщина связи |

### Правило уникальности

Связи считаются неориентированными парами:

```text
A → B == B → A
```

Поэтому второй вариант создать нельзя.

Канонический ключ пары строится функцией:

```js
canonicalLinkKey('A', 'B'); // A::B
canonicalLinkKey('B', 'A'); // A::B
```

### Ограничения

- ID узла не может быть пустым;
- ID узла не может повторяться;
- связь должна указывать на существующие узлы;
- узел нельзя связать с самим собой;
- одна пара узлов может иметь только одну связь.

## 🕸️ Узлы и связи

### Типы узлов

| Тип | Смысл по умолчанию | Стандартный размер |
|---|---|---:|
| `core` | Центральное ядро | 25 |
| `group` | Крупная ветвь или раздел | 18 |
| `node` | Обычный элемент | 13 |
| `accent` | Акцентный элемент | 16 |
| `default` | Нейтральный резервный тип | 14 |

Тип влияет только на оформление и начальную роль в раскладке. Он не навязывает предметный смысл.

### Порядок создания

1. Создать центральный узел `core`.
2. Создать группы первого уровня.
3. Создать дочерние узлы.
4. Создать связи.
5. При необходимости добавить дополнительные поперечные отношения.

### Пример

```js
graph.run('add_node', {
  id: 'core',
  name: 'Информационная система',
  type: 'core'
});

graph.run('add_node', {
  id: 'security',
  name: 'Информационная безопасность',
  type: 'group'
});

graph.run('add_connection', {
  from: 'core',
  to: 'security',
  label: 'включает'
});
```

## 🪐 Раскладки

### Планетарная паутина

Значение:

```js
layout.type = 'planetary';
```

Принцип работы:

1. Ядро ставится в центр.
2. Алгоритм строит дерево от ядра.
3. Узлы первого уровня распределяются по окружности.
4. Дочерние узлы продолжают направление своей ветви.
5. Глубина добавляет небольшое смещение по оси Z.
6. После построения выполняется устранение столкновений.

Основные параметры:

```js
layout: {
  planetary: {
    ringGap: 190,
    branchSpread: 0.56,
    depth: 34
  }
}
```

- `ringGap` — расстояние между уровнями;
- `branchSpread` — угловой разброс дочерних ветвей;
- `depth` — смещение по глубине.

### Желейные соты

Значение:

```js
layout.type = 'hex';
```

Принцип работы:

1. Узлы сортируются по уровню и ID.
2. Генерируется аксиальная спираль шестигранной сетки.
3. Координаты `q/r` переводятся в экранные X/Y.
4. Чётные и нечётные кольца получают небольшую глубину Z.
5. Между соседними позициями создаются пружины.
6. Импульс одного узла распространяется по физической сети.

Основные параметры:

```js
layout: {
  hex: {
    gap: 112,
    depth: 14
  }
}
```

## 🧲 Физика и движение

Физика работает как комбинация сил.

### 1. Якорная сила

Возвращает узел к рассчитанной позицией раскладки.

```text
F_anchor = (target - position) × anchorStrength
```

### 2. Пружина связи

Удерживает связанные узлы на разумном расстоянии.

```text
F_spring = displacement × springStrength
```

### 3. Отталкивание

Не позволяет узлам слипаться.

```text
F_repulsion ∝ repulsionStrength / distance²
```

### 4. Демпфирование

Постепенно уменьшает скорость.

```text
velocity *= exp(-damping × dt)
```

### 5. Ограничение скорости

Защищает симуляцию от взрыва координат при большом импульсе.

### 6. Подшаги

Один визуальный кадр делится на несколько физических вычислений:

```js
physics.substeps = 3;
```

Это делает движение устойчивее при нестабильном FPS.

### Камера

Камера хранит:

- текущий угол X;
- текущий угол Y;
- целевой угол X;
- целевой угол Y;
- угловые скорости;
- текущий zoom;
- целевой zoom.

Из-за разделения текущих и целевых значений движение получается плавным.

## ❤️ Органическая пульсация

Сердечный цикл формируется функцией `heartBeat()`.

Он состоит из трёх гауссовых импульсов:

1. сильный первый удар;
2. более слабый второй удар;
3. мягкая фаза восстановления.

Упрощённо:

```text
beat = first + second × 0.62 + recovery × 0.12
```

### Передача по уровням

Для каждого узла определяется уровень от ядра.

```text
phase(node) = time × beatsPerSecond - level × branchDelay
```

Поэтому:

- ядро сокращается первым;
- узлы первого уровня реагируют позже;
- второй уровень получает ещё большую задержку;
- импульс визуально проходит по ветвям.

### Органическая форма

При стиле `organic` контур узла строится не идеальным кругом. Радиус каждой точки немного изменяется синусоидальными волнами.

Результат:

- узел «дышит»;
- форма мягко меняется;
- свечение не требуется;
- эффект воспринимается как физическое сокращение, а не мигающая лампочка.

### Настройка

```js
networkPulse: {
  enabled: true,
  style: 'organic',
  bpm: 62,
  branchDelay: 0.115,
  travelWindow: 0.48,
  nodeAmplitude: 0.15,
  nodeDeformation: 0.06,
  linkWidthBoost: 1.35,
  glowEnabled: false,
  markerSize: 3.8
}
```

## 💨 Поток по ветвям

Постоянный поток отделён от сердечного импульса.

Сердечный импульс показывает периодическую волну, а `links.flow` создаёт непрерывное движение.

```js
links: {
  flow: {
    enabled: true,
    count: 4,
    speed: 0.28,
    size: 2.2,
    opacity: 0.92,
    trailLength: 0.085,
    shape: 'streak'
  }
}
```

Каждая частица:

1. получает собственное смещение по фазе;
2. движется от начала связи к концу;
3. проходит easing-функцию;
4. рисуется короткой линией с градиентным хвостом;
5. не имеет круглой головы, если `headDot: false`.

## 📐 Сужающиеся связи

Обычная Canvas-линия имеет одинаковую толщину везде. В Graph Studio связь строится как замкнутая лента.

### Геометрия

Для двух узлов вычисляются:

- единичный вектор вдоль связи;
- перпендикулярный вектор;
- начало после отступа от первого узла;
- конец до границы второго узла;
- центральная точка;
- ширина в начале;
- ширина в середине;
- ширина в конце.

### Ограничение конца

```text
endpointWidth = min(middleWidth, nodeRadius × endpointRatio)
```

То есть связь может быть сильной в центре, но становится тоньше у маленького узла.

### Настройка

```js
links: {
  taper: {
    enabled: true,
    endpointRatio: 0.34,
    minEndpointWidth: 0.55,
    insetRatio: 0.82
  }
}
```

## 👁️ Режим просмотра

Graph Studio является модулем, поэтому публичный просмотр рекомендуется встраивать как Web Component или iframe.

В режиме `viewer`:

- административная панель отсутствует;
- узлы нельзя изменять;
- связи нельзя изменять;
- команда записи блокируется;
- редактор связи не открывается;
- камера и информационные карточки работают.

Пример:

```html
<graph-studio-widget
  src="/data/project.json"
  mode="viewer"
  height="620px"
  controls>
</graph-studio-widget>
```

### Важное ограничение

Viewer-режим защищает обычный интерфейс, но не заменяет серверную авторизацию. Если данные записываются через backend, права доступа обязан проверять сервер.

## 🕸️ Graph Studio v8: полное техническое описание программы

Этот документ объясняет не только возможности программы, но и полный путь данных: от HTML-формы и JSON до физических координат, Canvas-проекции, событий, Web Component и MCP-sidecar.

## 1. Назначение системы

Graph Studio состоит из четырёх логических продуктов, использующих одно ядро:

1. **Административный редактор** для ручного создания графа.
2. **Движок визуализации** для отрисовки и интеракции.
3. **Встраиваемый viewer** для размещения на другом сайте.
4. **Машинный интерфейс** для JSON-планов, команд и MCP.

Ни один из этих режимов не должен иметь отдельную копию логики графа.

import './src/ui/app-controller.js';

Это намеренно. Корневой файл остаётся стабильной точкой входа, а реализация находится в модуле UI.

refreshEditor()

## 3. Жизненный цикл движка

### Создание

```js
const engine = new Graph3DEngine({ canvas, config, data });
```

Конструктор создаёт:

- `ctx`;
- сериализуемые `data.nodes/data.links`;
- сценовые `nodes/links`;
- быстрый `nodeMap`;
- списки соседей;
- виртуальные пружины;
- частицы;
- камеру;
- указатель;
- hover-состояние;
- таймеры анимации;
- `AbortController`;
- `ResizeObserver`.

### Запуск

`start()`:

- проверяет, не уничтожен ли движок;
- снимает паузу;
- запоминает время;
- вызывает `requestAnimationFrame()`;
- отправляет событие состояния.

### Кадр

Каждый кадр проходит последовательность:

```text
frame(timestamp)
  ├─ вычислить dt
  ├─ update(dt)
  │   ├─ updateCamera(dt)
  │   ├─ updatePhysics(dt)
  │   └─ updateProjection()
  ├─ renderOnce()
  │   ├─ drawBackground()
  │   ├─ drawParticles()
  │   ├─ drawLinks()
  │   ├─ drawNodes()
  │   └─ drawTooltip()
  └─ requestAnimationFrame(frame)
```

### Уничтожение

`destroy()`:

- ставит паузу;
- помечает экземпляр уничтоженным;
- отменяет события;
- отключает ResizeObserver;
- очищает массивы и Map.

12. обновить проекцию;
13. нарисовать кадр.

## 6. Планетарная раскладка

Модуль `layouts.js` сначала строит дерево от ядра.

### Уровни

BFS создаёт карту:

```text
core      → level 0
group     → level 1
node      → level 2
```

### Углы

Узлы первого уровня делят полный круг.

Дочерний узел получает угол относительно направления родительской ветви.

### Радиус

```text
radius = level × ringGap
```

### Глубина

Небольшое Z-смещение делает плоскую паутину объёмной после перспективного поворота.

---

## 7. Hex-раскладка

Hex-раскладка использует аксиальные координаты `q/r`.

Преобразование:

```text
x = gap × √3 × (q + r/2)
y = gap × 1.5 × r
```

Координаты генерируются спиралью по кольцам.

Пружины hex-сетки создаются между физически близкими позициями, даже если между узлами нет пользовательской связи. Благодаря этому сетка реагирует целиком.

---

## 8. Камера

Canvas остаётся двумерным, но каждый узел имеет X/Y/Z.

### Поворот Y

```text
x1 = x cosY - z sinY
z1 = z cosY + x sinY
```

### Поворот X

```text
y2 = y cosX - z1 sinX
z2 = z1 cosX + y sinX
```

### Перспектива

```text
scale = focalLength / (focalLength + z)
```

Экранные координаты:

```text
screenX = centerX + x × scale × zoom
screenY = centerY + y × scale × zoom
```

`nearClip` не позволяет знаменателю стать слишком маленьким.

---

## 9. Инерция камеры

При перетаскивании фона рассчитывается угловая скорость.

После отпускания:

```text
velocity *= exp(-friction × dt)
angle += velocity × dt
```

Параллельно текущие углы приближаются к целевым через сглаживание.

Автоматическое вращение добавляет постоянную скорость только при включённом `autoRotate`.

---

## 10. Пружинная физика

Для каждого узла вычисляется суммарное ускорение.

### Anchor

Возвращает узел к позиции раскладки.

### Link spring

Передаёт движение вдоль реальной связи.

### Hex spring

Связывает соседние соты.

### Repulsion

Раздвигает узлы.

### Damping

Поглощает энергию, чтобы система не колебалась бесконечно.

### Collision

Не позволяет центрам оказаться ближе суммы радиусов с padding.

### Propagated impulse

При воздействии на узел BFS распространяет импульс до `impulseDepth` уровней, уменьшая его через `impulseFalloff`.

---

## 11. Сердечная волна

`heart-pulse.js` является чистым математическим модулем.

### `gaussian()`

Создаёт мягкий импульс вокруг центра.

### `heartBeat()`

Складывает первый удар, второй удар и восстановление.

### `nodeHeartWave()`

Сдвигает фазу по уровню узла.

### `linkHeartWave()`

Возвращает:

```js
{
  intensity,
  progress
}
```

- `intensity` влияет на толщину и деформацию;
- `progress` задаёт положение импульса на связи.

---

## 12. Органические узлы

`organic-shapes.js` строит путь Canvas по окружности.

Радиус каждой точки немного меняется:

```text
r(angle) = baseRadius × (1 + wave1 + wave2 + pulse)
```

Несколько синусоид с разными частотами устраняют механически правильный круг.

Интенсивность ограничивается, чтобы узел не разрывался визуально.

---

## 13. Сужающаяся связь

`buildLinkRibbonGeometry()` создаёт чистую геометрию.

### Сначала

Вычисляется направление и нормаль.

### Затем

Начало и конец сдвигаются к границам узлов.

### Потом

Вычисляются три ширины:

- sourceWidth;
- middleWidth;
- targetWidth.

### Рендер

`drawTaperedRibbon()` строит замкнутый путь двумя quadratic curve.

---

## 14. Потоковые штрихи

`drawFlowStreak()` получает `progress` от 0 до 1.

Голова:

```text
head = start + (end - start) × progress
```

Хвост:

```text
tailProgress = max(0, progress - trailLength)
```

Градиент становится прозрачным к хвосту.

---

## 15. Hit-test связи

Пользователь не может навести курсор на математическую линию толщиной 1 пиксель надёжно.

Поэтому `findLinkAtPoint()`:

1. перебирает видимые связи;
2. вычисляет расстояние от курсора до отрезка;
3. сравнивает его с hover threshold и толщиной;
4. возвращает ближайшую связь.

## 16. Контроллеры оверлеев

### OverlayManager

Хранит зарегистрированные окна и группы взаимного исключения.

Он рассчитывает варианты размещения:

- слева;
- справа;
- сверху;
- снизу.

Для каждого варианта вычисляет пересечение с:

- toolbar;
- legend;
- status bar;
- границами сцены;
- другими исключающими областями.

Выбирается вариант с наименьшим конфликтом.

### NodeInfoController

Подписывается на `graph:hoverchange` и показывает DOM-карточку узла.

### LinkEditorController

В admin-режиме показывает редактор связи.

При вводе:

- preview применяется временно;
- save фиксирует;
- cancel возвращает snapshot;
- remove удаляет связь.

## 17. LivePreviewController

Контроллер хранит committed snapshot каждого раздела.

### Preview

```js
preview(section, patch)
```

- сохраняет исходную конфигурацию;
- объединяет черновой patch;
- вызывает `updateConfig(..., { transient: true })`;
- помечает раздел dirty.

### Commit

- применяет последний patch без transient;
- обновляет committed snapshot;
- снимает dirty.

### Rollback

- возвращает сохранённый snapshot;
- снимает dirty.

## 🧰 Graph Studio v8: API-справочник

## 1. Импорт

```js
import {
  Graph3DEngine,
  DEFAULT_GRAPH_CONFIG,
  validateGraphData,
  layoutGraph,
  projectPoint3D
} from './3d-graph-engine.js';
```

## 2. Создание `Graph3DEngine`

```js
const engine = new Graph3DEngine({
  canvas,
  config,
  data
});
```

| Параметр | Тип | Обязательный | Назначение |
|---|---|---:|---|
| `canvas` | `HTMLCanvasElement` | да | Поверхность Canvas 2D |
| `config` | object | нет | Частичная конфигурация поверх `DEFAULT_GRAPH_CONFIG` |
| `data` | object | нет | `{ nodes, links }` |

Конструктор:

1. проверяет Canvas;
2. получает контекст 2D;
3. объединяет конфигурацию;
4. создаёт состояние камеры;
5. создаёт состояние указателя;
6. подключает `ResizeObserver`;
7. подключает события;
8. при наличии данных вызывает `setData()`.

---

## 3. Методы данных

### `setData(data, options?)`

Полностью заменяет узлы и связи.

```js
engine.setData(
  { nodes, links },
  {
    force: false,
    transient: false
  }
);
```

- `force` обходит блокировку редактирования. Используется внутренними административными или embed-механизмами.
- `transient` помечает изменение как черновой предпросмотр.
- Перед записью вызывается `validateGraphData()`.
- После записи вызывается `rebuildScene(true)`.
- Затем отправляется `graph:datachange`.

### `addNode(node, parentId?)`

```js
engine.addNode({
  id: 'node_1',
  name: 'Новый узел',
  type: 'node'
});
```

Если передать `parentId`, метод одновременно создаст связь с родителем.

### `updateNode(id, patch)`

```js
engine.updateNode('node_1', {
  name: 'Изменённое название',
  color: '#556677'
});
```

ID сохраняется неизменным.

### `removeNode(id)`

Удаляет узел и все связи, в которых он является `source` или `target`.

---

## 4. Методы связей

### `addLink(link, options?)`

```js
engine.addLink({
  source: 'core',
  target: 'node_1',
  label: 'содержит',
  width: 4
});
```

Метод запрещает:

- связь с самим собой;
- отсутствующие узлы;
- повтор пары;
- обратный дубль.

### `updateLink(source, target, patch, options?)`

```js
engine.updateLink('core', 'node_1', {
  label: 'включает',
  color: '#778899',
  width: 6
});
```

Поиск выполняется независимо от направления пары.

### `removeLink(source, target, options?)`

```js
engine.removeLink('core', 'node_1');
```

---

## 5. Конфигурация

### `setConfig(config, options?)`

Полностью формирует конфигурацию заново поверх значений по умолчанию.

```js
engine.setConfig(config, {
  preserveCamera: true,
  force: false,
  transient: false
});
```

### `updateConfig(patch, options?)`

Частично обновляет конфигурацию.

```js
engine.updateConfig({
  particles: { enabled: false },
  camera: { autoRotate: true }
}, {
  rebuild: 'auto',
  force: false,
  transient: false
});
```

`rebuild`:

- `true` — всегда перестраивать сцену;
- `false` — не перестраивать;
- `'auto'` — перестраивать при изменении `layout`, `physics`, `node`, `colors`.

### `setLayout(type)`

```js
engine.setLayout('planetary');
engine.setLayout('hex');
```

Legacy-значение `grid` нормализуется в `hex`.

## 6. Блокировка

### `setEditingLocked(locked)`

```js
engine.setEditingLocked(true);
```

При блокировке:

- методы изменения выбрасывают ошибку;
- перетаскивание узла прекращается;

## 8. Анимационный цикл

### `start()`

Запускает render loop и снимает паузу.

### `pause()`

Останавливает запрос новых кадров, сохраняя сцену.

### `resume()`

Продолжает цикл.

### `togglePause()`

Меняет состояние паузы.

### `renderOnce()`

Рисует один кадр. Используется при изменениях в остановленном состоянии.

### `destroy()`

Обязательно вызывать при удалении виджета.

Метод:

- останавливает цикл;

## ⚙️ Graph Studio v8: полный справочник конфигурации

Этот документ описывает каждый параметр `DEFAULT_GRAPH_CONFIG`. Канонический источник находится в `src/core/default-config.js`.

- Для частичного изменения используйте `engine.updateConfig(patch)`.

## Замечание о единицах

- Углы движка хранятся в радианах. UI может показывать градусы.
- Скорости физики рассчитываются относительно секунд.
- Размеры Canvas задаются в CSS-пикселях, а backing store учитывает DPR.
- Цвета узлов и связей ожидаются в формате CSS; машинные схемы обычно ограничивают индивидуальные цвета форматом `#RRGGBB`.

## 🧠 Graph Studio v8: форматы данных

## 1. Внутренний формат движка

```json
{
  "nodes": [],
  "links": []
}
```

### Узел

```json
{
  "id": "core",
  "name": "Главная тема",
  "type": "core",
  "description": "Подробности",
  "color": "#263846",
  "size": 25
}
```

#### Ограничения

- `id` обязателен;
- `id` должен быть строкой;
- `id` не должен быть пустым;
- `id` должен быть уникальным;
- рекомендованный шаблон ID: `[A-Za-zА-Яа-яЁё0-9_-]+`;
- `size` рекомендуется держать в диапазоне 3–100;
- `color` для машинного ввода ожидается как `#RRGGBB`.

### Связь

```json
{
  "source": "core",
  "target": "branch",
  "label": "содержит",
  "description": "Связь главной темы с разделом",
  "color": "#8699a7",
  "width": 3
}
```

#### Ограничения

- `source` и `target` обязательны;
- оба ID должны существовать;
- `source` не может равняться `target`;
- пара узлов должна быть уникальной;
- направление не создаёт новую пару;
- ширина рекомендуется в диапазоне 0.5–20.

## 7. postMessage

Запрос:

```json
{
  "channel": "graph-studio",
  "widgetId": "my-graph",
  "requestId": "req-1",
  "tool": "get_project",
  "arguments": {}
}
```

Ответ:

```json
{
  "channel": "graph-studio",
  "widgetId": "my-graph",
  "requestId": "req-1",
  "ok": true,
  "result": {}
}
```

Ошибка:

```json
{
  "channel": "graph-studio",
  "widgetId": "my-graph",
  "requestId": "req-1",
  "ok": false,
  "error": "Описание ошибки"
}
```

Для приёма сообщений Web Component требует явный `allowed-origin`.
