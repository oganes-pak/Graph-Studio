# 🕸️ Graph Studio v8: полное техническое описание программы

Этот документ объясняет не только возможности программы, но и полный путь данных: от HTML-формы и JSON до физических координат, Canvas-проекции, событий, Web Component и MCP-sidecar.

## 1. Назначение системы

Graph Studio состоит из четырёх логических продуктов, использующих одно ядро:

1. **Административный редактор** для ручного создания графа.
2. **Движок визуализации** для отрисовки и интеракции.
3. **Встраиваемый viewer** для размещения на другом сайте.
4. **Машинный интерфейс** для JSON-планов, команд и MCP.

Ни один из этих режимов не должен иметь отдельную копию логики графа.

## 2. Как запускается приложение

### Шаг 1. Браузер открывает `index.html`

HTML создаёт:

- административную панель;
- вкладки;
- формы узла и связи;
- поля настройки движения;
- поля оформления;
- редактор легенды;
- JSON-консоль;
- Canvas;
- DOM-карточку узла;
- DOM-редактор связи;
- status bar.

### Шаг 2. Загружается `styles.css`

CSS:

- делит экран на редактор и сцену;
- ограничивает размеры панели;
- предотвращает наложение основных блоков;
- задаёт адаптивность;
- оформляет всплывающие карточки;
- задаёт состояния disabled/active/draft.

### Шаг 3. Загружается `editor.js`

Файл содержит только:

```js
import './src/ui/app-controller.js';
```

Это намеренно. Корневой файл остаётся стабильной точкой входа, а реализация находится в модуле UI.

### Шаг 4. Выполняется `app-controller.js`

Контроллер:

1. импортирует зависимости;
2. находит DOM-элементы;
3. объединяет начальную конфигурацию;
4. создаёт `Graph3DEngine`;
5. создаёт контроллеры легенды и оверлеев;
6. подключает формы;
7. публикует `window.graph`;
8. загружает проект;
9. запускает движок.

### Шаг 5. `boot()` загружает данные

Порядок:

```text
loadProjectSource()
       ↓ если src отсутствует
loadLocalProject()
       ↓
livePreview.syncCommitted()
       ↓
fillForms()
       ↓
refreshEditor()
       ↓
setAppMode('admin')
       ↓
engine.start()
```

## 3. Жизненный цикл движка

### Создание

```js
const engine = new Graph3DEngine({ canvas, config, data });
```

Конструктор создаёт:

- `ctx`;
- сериализуемые `data.nodes/data.links`;
- сценовые `nodes/links`;
- быстрый `nodeMap`;
- списки соседей;
- виртуальные пружины;
- частицы;
- камеру;
- указатель;
- hover-состояние;
- таймеры анимации;
- `AbortController`;
- `ResizeObserver`.

### Запуск

`start()`:

- проверяет, не уничтожен ли движок;
- снимает паузу;
- запоминает время;
- вызывает `requestAnimationFrame()`;
- отправляет событие состояния.

### Кадр

Каждый кадр проходит последовательность:

```text
frame(timestamp)
  ├─ вычислить dt
  ├─ update(dt)
  │   ├─ updateCamera(dt)
  │   ├─ updatePhysics(dt)
  │   └─ updateProjection()
  ├─ renderOnce()
  │   ├─ drawBackground()
  │   ├─ drawParticles()
  │   ├─ drawLinks()
  │   ├─ drawNodes()
  │   └─ drawTooltip()
  └─ requestAnimationFrame(frame)
```

### Уничтожение

`destroy()`:

- ставит паузу;
- помечает экземпляр уничтоженным;
- отменяет события;
- отключает ResizeObserver;
- очищает массивы и Map.

## 4. Разница между `data` и сценой

Движок хранит две формы графа.

### Сериализуемая форма

```js
engine.data = {
  nodes: [],
  links: []
};
```

Она предназначена для:

- экспорта;
- LocalStorage;
- JSON;
- MCP;
- Web Component;
- восстановления проекта.

### Сценовая форма

```js
engine.nodes
engine.links
```

Она дополнительно содержит:

- X/Y/Z;
- целевые координаты;
- скорость;
- экранные координаты;
- масштаб перспективы;
- уровень от ядра;
- ссылки на другие объекты;
- цвет после нормализации;
- вычисленный размер.

Сценовые поля не должны попадать в пользовательский JSON.

## 5. Перестройка сцены

`rebuildScene()` выполняется после изменения данных, раскладки, размеров или физики.

Основные шаги:

1. определить ядро;
2. вычислить уровни;
3. нормализовать типы узлов;
4. вычислить layout-позиции;
5. устранить начальные столкновения;
6. создать сценовые узлы;
7. создать Map по ID;
8. разрешить связи в ссылки на узлы;
9. построить adjacency;
10. создать spring pairs;
11. создать частицы;
12. обновить проекцию;
13. нарисовать кадр.

## 6. Планетарная раскладка

Модуль `layouts.js` сначала строит дерево от ядра.

### Уровни

BFS создаёт карту:

```text
core      → level 0
group     → level 1
node      → level 2
```

### Углы

Узлы первого уровня делят полный круг.

Дочерний узел получает угол относительно направления родительской ветви.

### Радиус

```text
radius = level × ringGap
```

### Глубина

Небольшое Z-смещение делает плоскую паутину объёмной после перспективного поворота.

---

## 7. Hex-раскладка

Hex-раскладка использует аксиальные координаты `q/r`.

Преобразование:

```text
x = gap × √3 × (q + r/2)
y = gap × 1.5 × r
```

Координаты генерируются спиралью по кольцам.

Пружины hex-сетки создаются между физически близкими позициями, даже если между узлами нет пользовательской связи. Благодаря этому сетка реагирует целиком.

---

## 8. Камера

Canvas остаётся двумерным, но каждый узел имеет X/Y/Z.

### Поворот Y

```text
x1 = x cosY - z sinY
z1 = z cosY + x sinY
```

### Поворот X

```text
y2 = y cosX - z1 sinX
z2 = z1 cosX + y sinX
```

### Перспектива

```text
scale = focalLength / (focalLength + z)
```

Экранные координаты:

```text
screenX = centerX + x × scale × zoom
screenY = centerY + y × scale × zoom
```

`nearClip` не позволяет знаменателю стать слишком маленьким.

---

## 9. Инерция камеры

При перетаскивании фона рассчитывается угловая скорость.

После отпускания:

```text
velocity *= exp(-friction × dt)
angle += velocity × dt
```

Параллельно текущие углы приближаются к целевым через сглаживание.

Автоматическое вращение добавляет постоянную скорость только при включённом `autoRotate`.

---

## 10. Пружинная физика

Для каждого узла вычисляется суммарное ускорение.

### Anchor

Возвращает узел к позиции раскладки.

### Link spring

Передаёт движение вдоль реальной связи.

### Hex spring

Связывает соседние соты.

### Repulsion

Раздвигает узлы.

### Damping

Поглощает энергию, чтобы система не колебалась бесконечно.

### Collision

Не позволяет центрам оказаться ближе суммы радиусов с padding.

### Propagated impulse

При воздействии на узел BFS распространяет импульс до `impulseDepth` уровней, уменьшая его через `impulseFalloff`.

---

## 11. Сердечная волна

`heart-pulse.js` является чистым математическим модулем.

### `gaussian()`

Создаёт мягкий импульс вокруг центра.

### `heartBeat()`

Складывает первый удар, второй удар и восстановление.

### `nodeHeartWave()`

Сдвигает фазу по уровню узла.

### `linkHeartWave()`

Возвращает:

```js
{
  intensity,
  progress
}
```

- `intensity` влияет на толщину и деформацию;
- `progress` задаёт положение импульса на связи.

---

## 12. Органические узлы

`organic-shapes.js` строит путь Canvas по окружности.

Радиус каждой точки немного меняется:

```text
r(angle) = baseRadius × (1 + wave1 + wave2 + pulse)
```

Несколько синусоид с разными частотами устраняют механически правильный круг.

Интенсивность ограничивается, чтобы узел не разрывался визуально.

---

## 13. Сужающаяся связь

`buildLinkRibbonGeometry()` создаёт чистую геометрию.

### Сначала

Вычисляется направление и нормаль.

### Затем

Начало и конец сдвигаются к границам узлов.

### Потом

Вычисляются три ширины:

- sourceWidth;
- middleWidth;
- targetWidth.

### Рендер

`drawTaperedRibbon()` строит замкнутый путь двумя quadratic curve.

---

## 14. Потоковые штрихи

`drawFlowStreak()` получает `progress` от 0 до 1.

Голова:

```text
head = start + (end - start) × progress
```

Хвост:

```text
tailProgress = max(0, progress - trailLength)
```

Градиент становится прозрачным к хвосту.

---

## 15. Hit-test связи

Пользователь не может навести курсор на математическую линию толщиной 1 пиксель надёжно.

Поэтому `findLinkAtPoint()`:

1. перебирает видимые связи;
2. вычисляет расстояние от курсора до отрезка;
3. сравнивает его с hover threshold и толщиной;
4. возвращает ближайшую связь.

---

## 16. Контроллеры оверлеев

### OverlayManager

Хранит зарегистрированные окна и группы взаимного исключения.

Он рассчитывает варианты размещения:

- слева;
- справа;
- сверху;
- снизу.

Для каждого варианта вычисляет пересечение с:

- toolbar;
- legend;
- status bar;
- границами сцены;
- другими исключающими областями.

Выбирается вариант с наименьшим конфликтом.

### NodeInfoController

Подписывается на `graph:hoverchange` и показывает DOM-карточку узла.

### LinkEditorController

В admin-режиме показывает редактор связи.

При вводе:

- preview применяется временно;
- save фиксирует;
- cancel возвращает snapshot;
- remove удаляет связь.

---

## 17. LivePreviewController

Контроллер хранит committed snapshot каждого раздела.

### Preview

```js
preview(section, patch)
```

- сохраняет исходную конфигурацию;
- объединяет черновой patch;
- вызывает `updateConfig(..., { transient: true })`;
- помечает раздел dirty.

### Commit

- применяет последний patch без transient;
- обновляет committed snapshot;
- снимает dirty.

### Rollback

- возвращает сохранённый snapshot;
- снимает dirty.

---

## 18. Валидация графа

`validateGraphData()` является центральным правилом.

Она проверяет:

- тип контейнера;
- массив nodes;
- массив links;
- каждый ID;
- уникальность ID;
- существование source/target;
- отсутствие self-loop;
- уникальность canonical pair;
- уникальность link ID.

Это правило должно применяться в браузере и серверном хранилище.

## 19. Атомарные команды

`GRAPH_TOOL_DEFINITIONS` отделяет описание команды от транспорта.

Каждая команда содержит:

```js
{
  name,
  title,
  description,
  inputSchema
}
```

`executeGraphTool()` преобразует простое имя в вызов движка.

Legacy aliases позволяют временно принимать старые MCP-имена.

---

## 20. JSON-план

`normalizeAiPlan()`:

1. проверяет объект;
2. проверяет версию формата;
3. нормализует узлы;
4. нормализует connections в links;
5. вызывает `validateGraphData()`;
6. переводит view в configPatch.

`applyAiPlan()` применяет данные и настройки.

`exportAiPlan()` выполняет обратное преобразование.

## 21. Viewer-policy

`access-policy.js` отделяет право чтения от права записи.

Viewer может запрашивать текущий проект, но не должен выполнять mutation tools.

UI также скрывает формы, но главным правилом остаётся проверка команды до выполнения.

## 22. Web Component

`GraphStudioWidget` наследуется от `HTMLElement`.

### Shadow DOM

Изолирует стили виджета от сайта.

### observedAttributes

Компонент реагирует на изменение:

```text
src
height
mode
locked
layout
controls
refresh-interval
```

### connectedCallback

Создаёт shell, движок, легенду, оверлеи и запускает загрузку.

### disconnectedCallback

Уничтожает таймеры, контроллеры и движок.

### postMessage

Команды принимаются только при совпадении `allowed-origin`.

## 23. MCP-sidecar

### `server.mjs`

Создаёт MCP-сервер и регистрирует:

- 13 tools;
- 4 resources;
- 2 prompts.

### `project-store.mjs`

Хранилище:

- создаёт файл при отсутствии;
- читает JSON;
- валидирует;
- создаёт draft;
- выполняет mutator;
- записывает во временный файл;
- атомарно переименовывает.

### Блокировка

`mutate()` отклоняет изменение, если `config.editor.locked === true`, кроме специальной команды снятия read-only.

## 24. Хранение в браузере

Административный редактор сохраняет проект в LocalStorage под ключом:

```text
graph-studio-v8-project
```

Transient-события не сохраняются.

При следующей загрузке проект восстанавливается, если внешний `src` не указан.

## 27. Что не следует делать

- Не помещать физику в обработчик кнопки.
- Не хранить второй независимый DEFAULT_CONFIG.
- Не менять `engine.nodes` из UI.
- Не создавать связь напрямую через `data.links.push()`.
- Не принимать AI JSON без валидации.
- Не открывать postMessage для `*`.
- Не писать служебные MCP-логи в stdout.
- Не считать скрытую кнопку полноценной авторизацией.
